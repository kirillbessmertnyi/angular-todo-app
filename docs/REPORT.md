## 2. Использованные подходы

### Page Object Model (POM)

Все локаторы и действия с UI сосредоточены в `tests/pages/todo.page.ts`. Тесты взаимодействуют только с методами `TodoPage`, не вызывая `page.locator()` напрямую. При изменении вёрстки достаточно обновить один файл.

Ключевое решение — метод `taskLocator(title)` использует `.filter({ has })` вместо CSS-интерполяции строк:
```typescript
taskLocator(title: string): Locator {
  return this.taskList.filter({
    has: this.page.locator('.task-title', { hasText: title }),
  });
}
```
Это безопасно для любых символов в заголовке задачи.

### Моки API через `ApiMock`

Класс `tests/helpers/api.mock.ts` перехватывает все запросы к `/todos**` через `page.route()`. Один обработчик диспетчеризует по HTTP-методу и структуре URL — GET / POST / PUT `:id` / DELETE `:id`. Колбэки `onPost`, `onPut`, `onDelete` позволяют тестам инспектировать тело запроса без глобальных побочных эффектов. `setup()` всегда вызывает `page.unroute()` перед регистрацией — исключает накопление обработчиков.

### Playwright Fixtures

Вместо `beforeEach` с ручной инициализацией применены кастомные фикстуры (`test.extend`) в `tests/fixtures.ts`:
- `apiMock` — создаёт экземпляр `ApiMock`.
- `todoPage` — зависит от `apiMock`, устанавливает заглушки и навигирует на `/` перед каждым тестом.

Зависимости теста явны в сигнатуре: `async ({ todoPage }) => { ... }`.

### Паттерн `Promise.all` для API-ассертов

Для проверки тела запросов использован атомарный паттерн:
```typescript
const [request] = await Promise.all([
  page.waitForRequest(req => req.url().includes('/todos') && req.method() === 'POST'),
  todoPage.addTask('My new task'),
]);
const body = await request.postDataJSON();
expect(body).toMatchObject({ title: 'My new task', completed: false });
```
Запрос захватывается одновременно с инициирующим действием — таймаут и порядок выполнения не влияют на результат.

### Assertions с автоповтором

Везде применяется `expect(locator).toHaveCount(n)` и `expect(locator).toBeVisible()` вместо `expect(await promise).toBe(n)`. Playwright автоматически повторяет `Locator`-assertions до таймаута — это устраняет флакинесс, вызванный асинхронным рендерингом Angular.

## 3. Обнаруженные баги

### BUG-07 — Редактирование задачи не сохраняется (P0, критический)

**Тест:** TC-04 · **Статус:** подтверждён на Chromium / Firefox / WebKit

**Симптом:** после ввода нового заголовка и нажатия Save задача сохраняет исходный заголовок.

**Причина:** шаблон использует одностороннее связывание `[value]="currentTask?.title"`. На каждый цикл Change Detection Angular перезаписывает `input.value` значением из компонента, затирая то, что напечатал пользователь. К моменту нажатия Save в `editInput.value` снова оказывается оригинальный заголовок.

**Подтверждение:** TC-13 перехватывает PUT-запрос и видит в теле исходный заголовок (а не отредактированный), что доказывает: проблема на уровне шаблона, а не сервиса.

**Рекомендация:** заменить `[value]` на `[(ngModel)]` (двустороннее связывание) или добавить обработчик `(input)`, пишущий в свойство компонента.

---

### BUG-04 — Нет визуального сообщения об ошибке при ответе сервера 500 (P1, major)

**Тест:** TC-17 · **Статус:** подтверждён на Chromium / Firefox / WebKit

**Симптом:** при ответе API со статусом 500 пользователь не получает никакого сообщения — ошибка поглощается молча.

**Причина:** подписки на `Observable` в методах `addTask`, `updateTask`, `deleteTask` содержат только `next`-колбэк без `error`-обработчика. При ошибке RxJS передаёт исключение в глобальный обработчик zone.js; компонент не обновляет состояние, элемент `.error` никогда не рендерится.

**Рекомендация:** добавить `error`-колбэк к каждой `subscribe()` и вывести сообщение через компонентную переменную + шаблон.

## 5. Сложности, возникшие при выполнении

### Нестабильность внешнего API

JSONPlaceholder принимает мутации, но не сохраняет данные. Первоначальная версия тестов работала с реальным API — результаты зависели от сетевой задержки и лимитов сервиса. Решение: полное замещение реального API через `page.route()` в классе `ApiMock`. Это сделало тесты детерминированными и не требующими сети.

### BUG-07: долгая диагностика

TC-04 падал с таймаутом, но TC-13 (проверка тела PUT-запроса) проходил. На первый взгляд это противоречие: PUT уходит — значит, Save работает. Потребовалось изучить шаблон компонента и понять механику Angular Change Detection, чтобы установить, что `[value]`-биндинг перезаписывает значение ввода до захвата данных компонентом. Диагностику упростило то, что TC-13 перехватывает тело запроса и видит в нём исходный заголовок.

### Порядок инициализации в TC-11

`todoPage`-фикстура автоматически навигирует на `/` с пустым списком задач. TC-11 проверяет начальную загрузку с двумя задачами из API — фикстура не подходила, так как моки нужно установить *до* навигации. Решение: TC-11 использует «сырые» фикстуры `page` + `apiMock` напрямую, а не `todoPage`, что позволяет явно управлять порядком `setup → goto`.
