# Angular ToDo App — E2E Test Suite

An Angular 16 to-do list application backed by the public [JSONPlaceholder](https://jsonplaceholder.typicode.com/) API, with a full Playwright E2E test suite.

---

## Prerequisites

| Tool | Version |
|------|---------|
| Node.js | 18 LTS or later |
| npm | 9+ |

---

## Setup

```bash
# Install project dependencies
npm install

# Install Playwright browsers (first-time only)
npx playwright install
```

---

## Running the Application

```bash
npm start
# Angular dev server starts at http://localhost:4200
```

---

## Running E2E Tests

Playwright automatically starts the Angular dev server before running tests (configured in `playwright.config.ts`).

```bash
# Run the full suite (headless, all browsers)
npm run test:e2e

# Run with the Playwright UI (interactive mode)
npm run test:e2e:ui

# Run a single spec file
npx playwright test tests/crud.spec.ts

# Run a specific test by title pattern
npx playwright test --grep "TC-01"

# Run on a single browser
npx playwright test --project=chromium

# Open the built-in Playwright HTML report
npm run test:e2e:report
```

## Allure Reports

[Allure](https://allurereport.org/) provides rich test reporting with history, steps, and screenshots.

### One-command workflow

```bash
# Run tests → generate report → open in browser
npm run allure
```

### Step by step

```bash
# 1. Run tests (generates raw XML into allure-results/)
npm run test:e2e

# 2. Build the HTML report from the raw results
npm run allure:generate

# 3. Open the generated report in the default browser
npm run allure:open

# Alternative: serve the live report directly from allure-results/
npm run allure:serve
```

### Output directories

| Directory | Contents | Committed |
|-----------|----------|-----------|
| `allure-results/` | Raw XML files generated by the reporter | No (`.gitignore`) |
| `allure-report/` | Rendered static HTML site | No (`.gitignore`) |

## CI/CD — GitHub Actions

The workflow file is at [.github/workflows/e2e.yml](.github/workflows/e2e.yml).

### What it does

```
push / PR / manual trigger
        │
        ├─ e2e (chromium)  ─┐
        ├─ e2e (firefox)   ─┼─ parallel matrix ──► allure-results-* artifacts
        └─ e2e (webkit)    ─┘
                              │
                         report job (always runs)
                              │
                    ┌─────────┴──────────────┐
                    │  download allure-history│  ← from previous run
                    │  merge browser results  │
                    │  allure generate        │
                    │  save allure-history    │  → for next run
                    └─────────┬──────────────┘
                              │
               ┌──────────────┴────────────────────┐
               │                                   │
        upload allure-report             write Job Summary
        artifact (30 days)           visible in Actions UI
        interactive HTML               without download
```

### Triggers

| Event | Tests run | Report available |
|-------|-----------|-----------------|
| Push to `main` | ✅ all 3 browsers | artifact + job summary |
| Pull Request | ✅ all 3 browsers | artifact + job summary |
| Manual (`workflow_dispatch`) | ✅ all 3 browsers | artifact + job summary |

### Viewing the report

**Job Summary — no download needed:**
Actions → select run → **Summary** tab

The summary shows a pass/fail table and a list of failed tests directly in the GitHub UI.

**Full interactive Allure report:**
Actions → select run → Artifacts section → download **`allure-report`** → serve locally (see below).

> ⚠️ **Do not open `index.html` directly from the filesystem.** Allure loads its data through `fetch()`, which browsers block on the `file://` protocol — the report will appear empty. Always use a local HTTP server.

```bash
# After unzipping allure-report into the project root:

# Option A — via allure CLI (recommended, opens browser automatically)
npm run allure:open

# Option B — via npm script (useful for a downloaded artifact)
npm run allure:serve-ci     # serves on http://localhost:8080

```

> No external hosting service is required. Everything works on free GitHub plans with public and private repositories.

### Allure history (trend graphs)

History is persisted via the `allure-history` artifact (retained 90 days). On each run the workflow downloads the previous history, passes it to `allure generate`, then saves the updated history. This populates **Trend**, **Retry**, and **Duration** graphs across runs without any external storage.

## Test Structure

```
tests/
├── helpers/
│   └── api.mock.ts          # ApiMock — deterministic route stubbing for /todos API
├── pages/
│   └── todo.page.ts         # Page Object Model for the ToDo UI
├── fixtures.ts              # Playwright custom fixtures (apiMock, todoPage)
├── crud.spec.ts             # TC-01–10: CRUD operations and filtering
├── api-behavior.spec.ts     # TC-11–17: API contract & resilience
└── validation.spec.ts       # TC-18–23: input validation & edge cases
```

### Test IDs

Every test title is prefixed with `TC-XX` which maps to the corresponding row in [docs/TEST_CASES.md](docs/TEST_CASES.md).

## Key Design Decisions

- **Page Object Model** — all selectors and actions live in `TodoPage`. When the UI changes, only that file needs updating.
- **`ApiMock` class** — a single `page.route()` handler covers GET/POST/PUT/DELETE. Tests call `apiMock.setup()` to install or override stubs without residual state.
- **Playwright fixtures** — `todoPage` fixture auto-stubs the API and navigates before each test; no boilerplate `beforeEach` in spec files.
- **`page.waitForRequest/Response()`** — used for API contract assertions instead of callbacks, eliminating timing races.
- **`expect(locator).toHaveCount()`** — preferred over `expect(promise).resolves.toBe()` to leverage Playwright's built-in auto-retry.

See [docs/QA_STRATEGY.md](docs/QA_STRATEGY.md) for the full rationale.
